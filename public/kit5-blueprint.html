Opportunity Brief
Why Queri, Why Now?

Every B2B SaaS product has the same invisible problem: users are constantly asking questions that the product itself could answer — but doesn't. "What plan am I on?" "Why can't I see this table?" "How do I set up an integration?" "Have I hit my limit?" Instead of getting an instant, contextual answer, users open a support ticket, search a docs site that doesn't know who they are, or give up entirely. The result: unnecessary churn, overwhelmed support teams, and a user experience that feels abandoned.

Queri closes this gap by combining RAG over your documentation with live user context from Stripe, your database, and your CRM — generating answers that are not just accurate about your product, but accurate about that specific user. It's the difference between "here's how billing works" and "you're on the Growth plan, you've used 80% of your query limit, and your next renewal is in 11 days."

The Context Gap No One Has Filled

Algolia Ask AI, Kapa.ai, Inkeep, and Mendable all solve the same problem: searching static documentation. They index your docs, apply RAG, and return a sourced answer. This is genuinely useful — but it covers only half the question surface area in a B2B SaaS product.

The other half of questions are context-dependent: they require knowing who the user is, what they've done, what they're entitled to, and what their current state is. No existing product connects these two layers. Queri does.

RAG + Context Architecture — 3 Layers

Layer 1 — Docs Ingestion (Async): Crawlee fetches all pages from connected docs sources. Each page chunked at 512 tokens with 64-token overlap. OpenAI embeddings generated per chunk. Stored in Supabase pgvector with metadata: source URL, page title, last indexed, tenant ID. Re-indexed on schedule (daily) or on webhook (doc updated).

Layer 2 — Live Context Assembly (Per-Query, <100ms): When a widget query arrives with a user ID, Queri fetches from cache (Redis, 5-min TTL): Stripe subscription plan + usage metrics + renewal date, app-specific context via the SaaS owner's configured REST hook (returns JSON: feature flags, current usage counts, permissions), HubSpot/Salesforce open deal or ticket status (optional). Assembled into a structured context block injected into the Claude prompt.

Layer 3 — Answer Generation (Claude Sonnet): System prompt defines strict rules: only cite what exists in the retrieved chunks or confirmed context. If unsure, say so. Never invent plan details or feature availability. Format answer in 2–4 sentences max for widget display. Include source link. If context shows user near a limit, include upgrade path contextually.
